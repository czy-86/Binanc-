# 🩺 深度大纲提取与逻辑诊断框架

在加载长文或短贴手册前，必须先使用此框架对原始素材进行“去水、刺破、搭桥”。

## §1 诊断阶段：冷血医生的三维刺破

在这个阶段，拒绝被素材中的“革命性、首创、极速”等形容词洗脑。

1.  **第一刺：找“行业疲惫点” (Pain Points Identification)**
    - 素材里在吹嘘什么？（如：我们很快）
    - 反向思考：为什么现在还没人能做到这么快？（物理成本太高、或者牺牲了安全性）
    - 结论：找到那个让用户感到“又来一个这种叙事”或“这个坑还没填好”的厌倦点。

2.  **第二刺：找“结构性摩擦” (Conflict Discovery)**
    - 核心矛盾 X vs Y：例如“极致隐私” vs “监管合规”；“极速性能” vs “草根节点参与度”。
    - 找出项目方在白皮书里试图“模糊化”或“轻描淡写”处理的那个代价。

3.  **第三刺：找“硬骨头” (The Hardbone)**
    - 对比那些发个模型、套个 UI 的“简答题”项目，这个项目最值得尊敬（或最笨拙）的那个“修硬轨”动作是什么？（如：重构了底层的签名协议、建立了一个极其复杂的链下证明系统）。

## §2 逻辑映射阶段：五步绝杀大纲

将诊断出的本质，精准填入以下“一镜到底”的逻辑链条中：

| 步骤 | 大纲要点 (Outline Point) | 逻辑诊断细节 (Cold Diagnosis) |
| :--- | :--- | :--- |
| **Step 1: 痛点** | **反高潮起手** | 描绘一个具体的“微观疲惫场景”（如：等确认时的那根烟、被夹子夹到麻木的情绪）。 |
| **Step 2: 类比** | **术语降维** | 把高深技术映射到“超市收银、玻璃房、修路费”等物理账本逻辑。重点是**谁给钱、谁给钱、利益怎么分**。 |
| **Step 3: 拆解** | **破壁人登场** | 引入项目。定义其身份：它不是在加功能，而是在“拆承重墙”或“修补结构性病灶”。 |
| **Step 4: 考验** | **拥抱摩擦力与重感** | 绝不谈理想乌托邦。引入物理学视角，指出项目在真实工程交付中遭遇的“粗糙感、混乱感、阻力与人性弱点”，赋予内容沉甸甸的真实感。 |
| **Step 5: 留白** | **冰冷观察** | 设定标准线。不喊单，只指出它成功的唯一死穴或唯一的变量 X。 |

## §3 数据锚点提取 (Extreme Parameters)

在大纲末尾，必须提取出至少一个具有“重金属感”的底牌参数：
- ❌ **错误**：我们的吞吐量很高。
- ✅ **正确**：在牺牲了 X 的前提下，它将跨链验证成本从 45U 压死到了 0.02U。

## §4 逻辑钩子校验 (👉 Causal Hook Check)

检查大纲的每一项之间是否存在推演的必然性（必须符合强因果逻辑）：
- [ ] 痛点是否引向了不平衡的旧规则？
- [ ] 类比是否引向了规则的失效？
- [ ] 主角是否作为“物理层面的破坏者”出现？
- [ ] 代价是否与主角带来的解药药量正相关？
